# trust-eats PRD v0.1

> 以人为维度的餐厅评价网络。不是大众点评的竞品，是 trust-based review network。
>
> v0.1 精简版，讨论后扩展。

---

## 1. 背景 & 问题

现有餐饮评价平台（大众点评、Google Maps、小红书）的根本缺陷：

- **单店维度，匿名众数**：平均分是所有用户等权平均，刷单号成本低——一家店雇 1000 个假账号打 5 星，平均分就上去。
- **打分人不可见**：米其林主厨的 8 分和从不吃辣游客的 8 分在系统里等权重，但对你完全没用。
- **数据归平台**：用户贡献内容是上传到平台，平台想怎么用怎么用。用户没有控制权，离开平台数据带不走。

核心问题：**trust 缺失**。平台靠治理维持 trust，但治理永远是攻防战。

---

## 2. 产品愿景

**trust-eats = 以人为维度的餐厅评价网络**

信任单位从"店"转移到"人"。一个用户的全部餐厅清单 + 评分 + 偏好画像作为一个整体被评估。

- 大众点评：你在某地，看这地有什么好吃的（店维度）
- trust-eats：你去某地玩，相信去过的人对**多家店**的点评，选自己想去的地方（人维度）

这不是点评竞品，是 trust network。

---

## 3. 目标用户

混合：

- **贡献者（Creator）**：本地吃货、旅行者，定期记录自己吃过的店，维护个人 profile
- **消费者（Reader）**：想去某地玩的人，查信任的人的清单，做选择

早期同一人可以同时是两种角色。冷启动阶段重点是 Creator（个人样板）。

**初期地域范围**：先香港，做透一个城市。

---

## 4. 核心概念

| 概念 | 说明 |
|---|---|
| Profile | 一个用户的全部身份：基本信息、品味画像、餐厅清单、评分历史 |
| 清单（List） | 一个 profile 下按主题/地区归类的店集合。如"中环早茶"、"我的 Top 10" |
| 店（Restaurant） | 一家店的元信息：地址、菜系、营业等。不含"平台评分" |
| 评价（Review） | 某用户对某店的打分 + 推荐/避雷菜 + 感受。归用户所有 |
| 信任关系 | followers / friends（见 §5） |
| 可见性（Visibility） | 每条数据可设 Private / Friends / Public |

---

## 5. Trust 机制

### 5.1 可见性三档

用户对自己所有数据（profile / 清单 / 评价）可选择三种可见性：

| 可见性 | 谁能看 | 系统能否拿去做推荐/聚合 |
|---|---|---|
| **Private** | 仅自己 | 否 |
| **Friends** | 双向好友 | 否（只走朋友间信任链，不走平台算法） |
| **Public** | 任何人 | 是 |

**关键区分：可见性 ≠ 可用性。** 你能看到好友的 friends-only 清单，不代表系统能把它推荐给陌生人。只有 Public 数据才进系统推荐池。

### 5.2 关系层级

- **单向 follow = 粉丝**：你关注某人，可看他的 public 内容。UI 标记：普通关注图标。
- **双向 follow = 朋友**：双方互关，可看 friends-only 内容。UI 标记：头像加绿点/特殊边框。

两层并存，用户清晰区分"我关注谁"和"谁是真朋友"。

### 5.3 Public 数据的用法

| 用法 | 是否做 | 说明 |
|---|---|---|
| **a. 推荐 profile** | ✅ v0.1 | 把 public 用户推荐给其他用户关注（"和你口味相近的人"）。推荐基于品味画像相似度，不是全网平均 |
| **b. 个人化加权聚合分** | ✅ v0.1 | 某家店的"聚合分" = 你关注的人 + 与你品味相似的人打的分的加权平均。同一家店你看到 8.6，你朋友看到 7.2，因为你们信任的人不同。**不做全网平均分**——这是和大众点评的核心区别 |
| **c. 外部 API** | ❌ v0.1 | 不开放外部消费。PRD 留扩展位，未来视用户量再决定 |

### 5.4 删除权

**数据主权最硬的体现：用户走的时候数据带走。**

- **Profile 删除**：用户删账号 → 系统级硬删，所有 public 评价从聚合分里消失，聚合分实时重算
- **单店评价删除/更新**：用户改主意 → 硬删旧评价（系统重算），但保留版本号供用户自己查看历史（"我曾于 X 年给 Y 分"）

刷单视角：硬删让刷单号"删号跑路"容易，但配合 §5.1 的"public profile 暴露全量数据"设计，刷单号本来就要长期伪装才能影响推荐，删号跑路等于放弃投入，不构成刷单工具。

### 5.5 数据导出

- **年报图**：用户可生成自己的"2025 美食年报"，一张可分享的图片，包含今年吃过店数、平均分、最爱菜系、地理分布等
- **不是 CSV/JSON 导出**（面向机器），而是面向社交分享的视觉化年报
- 具体设计（店多时长图怎么排版等）单做，不在 PRD v0.1 范围

## 6. MVP 用户场景

### 场景 1：朋友问"香港玩三天吃啥"

- 小明想去香港玩，问小红
- 小红是 trust-eats 用户，profile 设 public，维护了 50+ 家香港餐厅
- 小红把 profile 链接甩给小明，小明看到：小红 profile 主页、按地区归类的清单、每家店的打分和必点菜
- 小明还可以一键"关注小红"，下次小红更新清单自动出现在小明 feed
- **关键：不需要平台算法，朋友间直接走信任链**

### 场景 2：陌生人查某家店

- 小明刷到亚参海南鸡饭，想看 trust-eats 上评价
- 他打开 trust-eats，搜"亚参海南鸡饭"
- 看到：**不是全网平均分**，而是"你关注的人打 8.6（3 人）"、"和你品味相近的人打 8.4（12 人）"
- 两个分都可点击展开看具体是谁打的
- 如果小明没关注任何人、没品味画像 → 显示所有 public 用户的分数分布，但不显示单一"权威平均分"

### 场景 3：创作者记录

- 小红刚吃完一家店，打开 trust-eats 录入
- 填写：店名、地区、菜系、四项评分（味道/环境/服务/性价比）、推荐菜、避雷菜、感受
- 可选：上传照片
- 可见性：private / friends / public 三选一（默认 private）
- 提交 → 进入她的 profile + 自动加入对应清单 + 若 public 则进系统推荐池

### 场景 4：品味相似度推荐

- 小明关注了 3 个香港吃货，系统算出他品味画像："爱烧鹅、爱辣、不爱甜"
- 系统推荐：还有 5 个 public 用户的品味画像和小明相似度 > 0.8，要不要关注？
- 小明关注后，他的聚合分开始基于这 8 个人的加权评分

### 场景 5：删除账号

- 小红不想用了，点"删除账号"
- 系统二次确认 → 删除全部数据（profile / 清单 / 评价 / 信任关系）
- 她曾贡献的 public 评分从所有聚合分里消失，实时重算
- 她关注过的人的 friends 列表里她消失

- 用户注册 / 登录
- Profile 页：基本信息 + 品味画像 + 清单索引

- 按地区/菜系/主题归档清单
- 搜索店（看 public 评价 + 自己关注的人的评价）
- 个人化加权聚合分（基于关注的人 + 品味相似的人）

## 8. 抗刷单逻辑

**核心：刷单成本从"刷 1 家店"变成"伪装 1 个真人长期生活"。**

1. **Public profile 暴露全量数据**：要影响推荐，刷单号必须设 public，意味着它的全部清单、打分模式、画像暴露。空号一眼看穿，长期伪装成本极高。
2. **Friends-only 不进推荐池**：刷单号靠加好友刷 friends-only 数据无法影响陌生人推荐。
3. **个人化加权聚合**：没有"全网平均分"可刷。刷单号要影响你看到的分，必须先让你关注它，或让你的关注者关注它。两种都需要先建立 trust。
4. **硬删 + 重算**：刷单号被识别删除后，曾贡献的分实时消失，无法长期累积影响。

## 9. 冷启动策略

**先做个人样板，后扩社交圈。**

| 阶段 | 店数 | 用户数 | 目标 |
|---|---|---|---|
| 现在 | ~12 | 1（创始人） | 继续做厚个人 profile，验证数据结构 |
| 阶段 1 | 30-50 | 1 | 个人样板成型，作为 product showcase |
| 阶段 2 | 50-100 | 1 | 静态站点（GitHub Pages）让 profile 可被分享浏览 |
| 阶段 3 | 100+ | 5-10 | 邀请 5-10 个朋友加入，每人 20+ 家店，覆盖香港 |
| 阶段 4 | 多城市 | 多用户 | 产品化，开放注册 |

hk_meal 仓库作为创始人个人样板的数据源继续维护，trust-eats 是产品形态。

## 10. 开放问题

- 品味画像的具体算法（基于评分历史聚类？基于推荐菜/避雷菜关键词？）——v0.2 再定
- Friends 的上限有没有？关注有没有上限？——v0.1 不设，观察
- 店方账号怎么处理？店方是否能注册、是否能回应评价？——v0.3+
- 数据存储架构（用户数据是否真的"归用户"？是否走 federated / 本地优先？）——v0.2 设计
- 商业模式（完全免费？订阅？）——v0.1 不考虑，先验证产品
- 法律合规（评价版权、店方投诉、个人数据保护）——v0.2+
